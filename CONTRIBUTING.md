# Contributing

## 開発フロー

1. 対応するIssueで目的と受け入れ条件を確認する
2. 小さなブランチで変更する
3. `npm run package` で型チェック、テスト、バンドルを実行する
4. セキュリティ境界や外部送信先を変える変更は明記する
5. Pull Requestでは変更理由、利用者への影響、確認方法を説明する

## 実装方針

- Markdownを正本として保つ
- AI ProviderとRepository実装を交換可能にする
- AIが利用できない場合の手動経路を維持する
- 生成結果を自動確定せず、人が確認できるようにする
- 新しいメタデータは既存Markdownとの後方互換性を考慮する
- テレメトリーや外部通信は明示的な設計判断なしに追加しない

## Issueの粒度

1つのIssueは、検証可能な受け入れ条件を持つ小さな成果にします。将来案はロードマップへ残し、MVPの完成を妨げる依存関係を増やしすぎないようにします。
