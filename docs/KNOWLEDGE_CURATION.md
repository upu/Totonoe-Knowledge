# ナレッジ整理の設計コンセプト

Entryは変更点単位で積み上がるため、件数が増えると同じ対象への結論が複数Entryへ分散し、どれが現在有効か分からなくなります。このドキュメントは、v0.3〜v0.4で実装する整理機能の設計原則を定めます。

ナレッジは2層で保持します。

1. 粒ナレッジ: 調査・変更・決定ごとの個別Entry。登録の基本単位
2. Current View（累積版）: 粒ナレッジを統合し、対象の現在仕様をまとめたEntry

粒ナレッジは登録後も削除せず変更履歴として残し、累積版は粒ナレッジから承認付きで再構築できる統合結果として扱います。

## 原則: 整理は使用駆動で行う

全Entryを対象とした一括整理は行いません。数十件を超えた時点で定期的な全件見直しは続かず、矛盾や重複が実害になるのは、そのEntryが読まれる瞬間だけだからです。整理の機会は、人がすでに内容へ注意を向けている次の2時点へ埋め込みます。

1. 登録時: 新Entryと既存Entryの関係をその場で確定する
2. 検索時: 同時にヒットしたEntry間の不整合をその場で記録する

検索されないEntryは整理しません。使われる知識から順に品質が上がるため、全件を一度に片づけなくても、日常の利用そのものが整理を前進させます。

## 登録時: 関係候補の提示

新Entryの保存前に、既存Entryから類似上位数件を取得し、関係候補を提示します。

| 関係 | 意味 | 確定時の扱い |
|---|---|---|
| Duplicate | 同じ結論 | 登録を中止するか既存Entryを更新する |
| Supersede | 既存の結論を置き換える | 新Entryへ`supersedes`を設定する |
| Conflict | 両立しない結論が併存する | 双方へ`conflicts`を記録する |
| Complement | 同じ対象の別側面 | `related`を設定する |
| Related | 参照すると役立つ | `related`を設定する |

判定は構造で絞り込んでからAIで確認します。type、適用version範囲、keywords、embedding類似度で候補ペアを数件まで絞り、そのペアだけをLanguage Modelで分類します。全Entryペアの比較は行いません。AIは候補の提示までとし、関係の確定は人が行います。

候補検索は読み取り専用で行い、確認前にMarkdownや再生成可能な検索indexを更新しません。候補ごとに採用、関係種別の編集、却下を選び、新規Entryと既存Entryの変更差分を確認した後だけ、登録操作でまとめて反映します。途中でキャンセルした場合は新規・既存Markdownのどちらも変更しません。

類似候補にはCurrent View（累積版）も含めます。新しい粒ナレッジが既存の累積版の対象範囲に入る場合、その場で次のどちらかを選びます。

- すぐ反映する: 累積版の本文を更新し、`consolidates`へ新EntryのIDを追加する
- 後で反映する: 新Entryへ`affects: [<累積版ID>]`を記録して作業を続ける

登録の摩擦を増やさないため、候補の確認は省略できます。省略しても、同じ不整合は検索時に再び発見の機会があります。

## 検索時: 矛盾の記録

検索結果に、対象と適用範囲が重なり結論の異なるEntryが複数並んだとき、それが矛盾の発見です。その場で解決を強制せず、双方のEntryへ`conflicts: [<相手ID>]`を記録して作業を続けられるようにします。

`conflicts`が残っているEntryの一覧が、そのまま整理の作業キューです。別のTODOファイルや管理DBは作りません。front matterはGitでreviewでき、`Validate Repository`が未解消の`conflicts`を報告します。矛盾の解消とは、次のいずれかを行って`conflicts`を除去することです。

- 一方を`supersedes`で置き換える
- `applies_from` / `applies_to`で適用範囲を分ける
- 誤っているEntryの本文を修正する
- 両論併記が正しい場合は`related`へ変更し、判断根拠を本文へ書く

## Current View（累積版）: 統合はEntryの一種

累積した粒ナレッジから「現在の仕様」をまとめるCurrent Viewは、特別な生成物ではなく、`consolidates`を持つ通常のspecification Entryとして保存します。

```yaml
type: specification
consolidates:
  - KN-0012
  - KN-0031
  - KN-0058
consolidated_at: "2026-07-16T00:00:00+09:00"
```

- 本文はAIが元Entry群から下書きし、通常の登録フローで人が確認・編集して保存する
- 元Entryは削除せず変更履歴として残し、Evidenceへの追跡性を維持する
- `consolidates`（累積版→粒）と`affects`（粒→累積版）で双方向にたどれる。反映が完了したら、累積版の`consolidates`へIDを追加し、粒Entryの`affects`を除去する
- 再統合は同じ登録フローの繰り返しであり、Git履歴に差分が残る

次の条件を`Validate Repository`が検査し、累積版の陳腐化を警告します。検出は日時とfront matterの比較だけで行い、AIを使いません。

- `consolidates`内のEntryが`consolidated_at`より後に更新された
- `consolidates`内のEntryが、`consolidates`に含まれないEntryへ置き換えられた
- 未解消の`affects`がその累積版を指している

これにより「Markdownを正本とし、DBは再生成可能な派生物」という基本方針を崩さずに、変更点の累積から現行仕様を再構築できます。

## 品質の計測

品質は登録時のゲートを厳しくするのではなく、事後の計測で守ります。登録の摩擦は継続利用を壊すためです。

- 検索品質はdogfoodingで作成した回帰queryで守る。ランキングや索引の変更は、回帰queryの期待Entryが上位に残ることを合格条件とし、新しく見つかった検索の失敗例は回帰queryへ追加する
- `Validate Repository`を健全性レポートへ拡張し、未解消の`conflicts`、陳腐化したCurrent View、既存の整合性エラーを一覧する
- 検索後に開かれなかったEntryや長期間ヒットしないEntryの計測は検索logの保存を伴うため、必要性をdogfoodingで確認してから設計する
