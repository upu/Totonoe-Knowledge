# エージェント連携の設計コンセプト

自動検索と自動蒸留を、特定のAI製品に依存せず実現するための設計です。

会話ログ全量をベクトル化して毎回のプロンプトへ注入する記憶型の方式（RAG型長期メモリ）とは立場が異なります。Totonoe Knowledgeでは、会話ログはEvidence層であり、正本には人が承認した蒸留済みEntryだけを置きます。自動化するのは蒸留と検索であり、承認は自動化しません。

## 自動検索

Agentが人の指示なしにナレッジを参照するためのレバーは2つあります。

1. **Language Model Toolのdescription**。Agentモードの自動呼び出しは、Toolのdescriptionを読んだモデルの判断で起きます。`#totonoeKnowledgeSearch`のdescriptionには「このプロジェクト固有の仕様、過去の調査、既知の問題について確信がないとき、回答の前に検索する」という利用条件を明記し、dogfoodingで自動呼び出しの発生状況を見て文面を調整します。
2. **リポジトリ指示ファイル**。`.github/copilot-instructions.md`、`AGENTS.md`、`CLAUDE.md`などへ「このプロジェクトの仕様・経緯に関する質問へ答える前にナレッジ検索を使う」と記述します。拡張機能の実装を変えずに運用だけで試せるため、Tool descriptionの調整より先に検証できます。

どちらも、呼び出すかどうかの最終判断はモデル側にあります。呼ばれなかった検索は拡張機能から観測できないため、dogfoodingでは「検索していれば見つかったはずの質問」を人が記録して評価します。

## 自動蒸留: 受信箱パターン

セッション終了フックを持つエージェント（Claude Code等）、CI、社内バッチは、[prepared_knowledge形式](PREPARED_KNOWLEDGE.md)のMarkdownを受信箱フォルダーへ書き込むことで、ナレッジの下書きを非同期に供給できます。

```text
外部エージェント（セッション終了フック等）
  → 会話を要約し、prepared_knowledge形式で受信箱へ書く
  → 人が後でまとめてreview
  → AIを使わない登録フローで確認・保存（外部通信なし）
```

- 受け口はprepared_knowledge形式そのものです。エージェント側は決まった形式のMarkdownファイルを書くだけでよく、拡張機能のAPI、プロセス間通信、ネットワークを必要としません
- 受信箱は正本の外に置き（ナレッジリポジトリ外、または索引対象から除外したフォルダー）、未承認の内容が検索結果へ混ざらないようにします
- 登録時の人による確認、秘密情報候補検査、関係候補の提示（[KNOWLEDGE_CURATION.md](KNOWLEDGE_CURATION.md)）は省略しません

初期スコープ外とした「完全自動登録」との違いは、承認の位置です。蒸留の実行は自動化しても、正本への昇格は常に人が行います。

Codexの`SessionEnd`フックで候補を自動抽出する実装とセットアップは、[Codex受信箱フック](../examples/codex-inbox-hook/README.md)を参照してください。フックの3秒制約を守るため、終了処理ではジョブだけを作成し、別の読み取り専用`codex exec`ワーカーが候補を生成します。
