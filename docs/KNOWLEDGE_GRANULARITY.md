# Knowledge Entryの粒度ガイドライン

## 基本単位

1つのEntryには、後から独立して検索・更新・置き換えできる結論を1つだけ置きます。長さではなく、変更理由と有効期間が同じかで分割を判断します。

次の問いへ1つの短い答えを返せる粒度が目安です。

- 何が分かったか
- 何を決めたか
- 何が変わったか
- どの手順で再現・復旧するか
- どの仕様が有効か

背景、根拠、確認結果、手順、注意点は、その結論を再利用するために必要なら同じEntryへ含めます。

## 分割する条件

次のいずれかが異なる場合は、別Entryにします。

- 結論、決定、変更、手順の主目的
- 適用versionや有効期間
- owner、component、system境界
- 独立して置き換え・廃止される可能性
- reviewや公開範囲
- Knowledge type

単に本文が長い、同じ調査sessionで分かった、同じIssueに関係する、という理由だけではまとめません。

## まとめてよい条件

次は同じEntryにまとめます。

- 1つの結論を支える複数の確認結果
- 症状、原因、復旧方法、再発防止が一体になったtroubleshooting
- 1つのprocedureに必要な前提、手順、rollback、注意点
- 1つのdecisionに対する背景、比較案、採用理由、影響

## typeの選び方

| type | 中心となる内容 | 例 |
|---|---|---|
| `investigation` | 調査で判明した事実 | 性能低下の条件を特定した |
| `troubleshooting` | 症状から復旧・再発防止まで | 特定errorの原因と解消手順 |
| `specification` | 現在守るべき振る舞い | APIの入力制約と応答契約 |
| `change` | 以前から何が変わったか | version更新による設定変更 |
| `procedure` | 再実行可能な作業手順 | backupからのrestore手順 |
| `decision` | 選択とその理由 | Markdownを正本にする判断 |

複数typeに見える場合は、検索時に最初に知りたいものを選びます。調査結果から正式仕様や判断が確定した場合は、調査Entryを残し、別のspecificationまたはdecision Entryから関連付けます。

## title・summary・keywords

- titleは対象と結論を識別できる具体的な表現にする
- summaryは結論を1文で書き、背景や作業経過を入れない
- keywordsはtitleと同じ語の羅列ではなく、別名、製品名、error code、file名、検索時に使う表現を3〜7件程度選ぶ
- version、`supersedes`、関連IDは根拠が明示されている場合だけ設定する

## 時点情報

dogfooding記録や当時の調査結果は、後から実装が変わっても過去の事実として残します。現在仕様として再利用するEntryでは、次のどちらかを行います。

- 本文と`updated_at`を更新し、変更理由をGit履歴へ残す
- 旧Entryを残し、新Entryの`supersedes`で明示的に置き換える

異なるversionで両方が有効なら、`applies_from`と`applies_to`で範囲を分けます。

## 保存前チェック

- titleだけでEntryを区別できる
- summaryが1つの結論になっている
- 本文に独立した別の結論が混ざっていない
- typeと保存directoryが内容に合っている
- keywordsに後日使う別表現が含まれている
- 未確認事項を確定事項として書いていない
- 機密区分、AI送信先、Git公開範囲を確認した
- versionや置き換え関係を推測で設定していない

このガイドラインはv0.1の12件dogfoodingと、5件の内容品質再レビューを基にしています。意味的な重複候補の自動提示はv0.3のハイブリッド検索で扱います。
